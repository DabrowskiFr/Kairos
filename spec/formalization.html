Formalisation Abstraite de OBC vers OBC+

1. Objectif et portée

Ce document formalise le passage d'un noeud OBC source vers un noeud OBC+ enrichi
(cohérence des contrats, obligations monitor/compatibilité, obligations backend).

Avant d'énoncer le résultat final, on fixe les notations de haut niveau utilisées dans tout le texte.

Notations principales (domaine et rôle)

  • NNN : noeud source (programme OBC avant enrichissement).
  • N+N^+N+ : noeud enrichi (programme OBC+ après ajout d'obligations).
  • I\mathcal II : ensemble des variables d'entrée du noeud.
  • AAA : hypothèse de contrat (formule temporelle sur les entrées).
  • GGG : garantie de contrat (formule temporelle sur le comportement du noeud).
  • ρ\rhoρ : suite des états internes successifs du noeud.
    Intuition : mémoire + état de contrôle au fil des pas.
  • ι:N→StoreI\iota : \mathbb N \to Store_{\mathcal I}ι:N→StoreI​ : flux d'entrées.
    Intuition : valeur des entrées à chaque instant.
  • ω\omegaω : flux de sorties produit par le noeud.
  • π:=(ρ,ι,ω)\pi := (\rho,\iota,\omega)π:=(ρ,ι,ω) : exécution complète.
  • π↓I:=ι\pi\downarrow_{\mathcal I} := \iotaπ↓I​:=ι : projection de l'exécution sur les seules entrées.
  • π⊨Φ\pi \models \Phiπ⊨Φ : la formule temporelle Φ\PhiΦ est satisfaite par π\piπ.

Vue d'ensemble du langage OBC (intuition)

Un noeud OBC est un composant synchrone à états finis :

  • il lit des entrées à chaque pas d'horloge,
  • il met à jour son état interne,
  • il produit des sorties,
  • il choisit une transition de contrôle selon l'état courant (et éventuellement une garde).

Les contrats de transitions (require/ensure) expriment des propriétés locales d'un pas.
Les contrats LTL de noeud (AAA, GGG) expriment des propriétés temporelles globales sur toute l'exécution.

Ce que l'on veut vérifier

On veut établir que le noeud satisfait sa garantie temporelle GGG pour toute exécution dont
les entrées satisfont l'hypothèse AAA, c'est-à-dire :

∀π,  (π↓I⊨A)⇒(π⊨G).\forall \pi,\; (\pi\downarrow_{\mathcal I}\models A)\Rightarrow(\pi\models G).∀π,(π↓I​⊨A)⇒(π⊨G).

Intuitivement : sous les bons scénarios d'entrée, le comportement du programme reste toujours conforme au contrat.

Méthode de vérification (intuition)

La méthode suit trois idées :

  1. ramener le temporel à un monitor :
    compiler (A,G)(A,G)(A,G) en un automate monitor avec un état bad représentant la violation ;
  2. instrumenter le programme :
    ajouter des obligations de support (Coherency, Compatibility, Monitor) pour relier
    l'exécution du code à celle du monitor ;
  3. prouver les obligations générées :
    générer des VCs via WP ; la preuve de ces VCs établit notamment l'inatteignabilité de bad
    (NoBad), puis A⇒GA \Rightarrow GA⇒G via la correction du monitor (MonCorr).

Résultat visé

Le théorème de correction de bout en bout visé est :

∀π,  (π↓I⊨A)⇒(π⊨G).\forall \pi,\; (\pi\downarrow_{\mathcal I} \models A) \Rightarrow (\pi \models G).∀π,(π↓I​⊨A)⇒(π⊨G).

Lecture intuitive :
pour toute exécution possible du programme, si le flux d'entrées respecte l'hypothèse AAA,
alors la garantie GGG est satisfaite.

Intuition de la méthode (vue d'ensemble)

Avant les détails formels, l'idée globale est la suivante :

  1. on part des formules LTL de contrat (AAA, GGG) ;
  2. on construit un automate monitor M\mathcal MM qui encode la violation potentielle du contrat
    (avec un état bad) ;
  3. on instrumente le programme OBC avec une variable d'état monitor et des obligations logiques
    (Compatibility, Monitor, NoBad) qui forcent la cohérence entre exécution programme et
    exécution monitor ;
  4. on génère des obligations de preuve (WP/VC) sur ce programme instrumenté ;
  5. si ces obligations sont prouvées, on montre que l'état bad n'est jamais atteint ;
  6. par correction de la construction monitor, l'absence de bad sous l'hypothèse AAA
    implique la satisfaction de GGG.

En résumé : au lieu de prouver directement A⇒GA \Rightarrow GA⇒G sur le code brut, on prouve que
l'instrumentation monitor est respectée et que bad est inatteignable, ce qui donne GGG.

Rôle des obligations : objectif vs support

On distingue deux familles :

  • obligation objectif : NoBad (prouver que l'état monitor bad est inatteignable) ;
  • obligations de support : Coherency, Compatibility, Monitor (et obligations WP associées),
    qui servent à rendre la preuve de NoBad correcte et exploitable vis-à-vis de la sémantique du programme.

Le fil logique est :

Support prouveˊ  ⟹  NoBad prouveˊ  ⟹  (A⇒G) via MonCorr.\text{Support prouvé} \;\Longrightarrow\; \text{NoBad prouvé} \;\Longrightarrow\; (A \Rightarrow G) \text{ via } MonCorr.Support prouveˊ⟹NoBad prouveˊ⟹(A⇒G) via MonCorr.

Théorème cible (Tfinal)

Tfinal:∀π,  (π↓I⊨A)⇒(π⊨G).T_{final}:\quad \forall \pi,\; (\pi\downarrow_{\mathcal I} \models A) \Rightarrow (\pi \models G).Tfinal​:∀π,(π↓I​⊨A)⇒(π⊨G).

Plan de preuve (backward)

Pour prouver TfinalT_{final}Tfinal​, on isole les résultats intermédiaires suivants :

  • Préservation de l'exécution (LprojL_{proj}Lproj​) :
    pour toute exécution instrumentée π+\pi^+π+ de N+N^+N+, la projection
    π+↓N\pi^+\downarrow Nπ+↓N est une exécution valide de NNN (mêmes pas observables
    sur les variables du programme source).
  • Adéquation monitor (LmonL_{mon}Lmon​) :
    les obligations Compatibility/Monitor suffisent à relier AAA et GGG sur N+N^+N+.
  • Cohérence locale des contrats (LcohL_{coh}Lcoh​) :
    les obligations ajoutées propagent les require d'une transition vers la suivante.
  • Alignement temporel du shift (LshiftL_{shift}Lshift​) :
    ShiftIShift_{\mathcal I}ShiftI​ exprime correctement le passage du pas iii au pas i+1i+1i+1.
  • Définissabilité des historiques (LprekL_{prek}Lprek​) :
    les occurrences pre_k utilisées dans les obligations ne sont jamais évaluées hors domaine.

Chaîne de dépendance visée :

Lprek∧Lshift∧Lcoh∧Lmon∧Lproj  ⟹  Tfinal.L_{prek} \land L_{shift} \land L_{coh} \land L_{mon} \land L_{proj} \;\Longrightarrow\; T_{final}.Lprek​∧Lshift​∧Lcoh​∧Lmon​∧Lproj​⟹Tfinal​.
