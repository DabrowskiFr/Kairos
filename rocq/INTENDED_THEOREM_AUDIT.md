# Audit of the Intended Theorem

## Goal

This note is not a proof artifact. Its purpose is to answer a different
question:

- are we still proving the theorem we intended to prove?

The fact that Rocq still compiles is not sufficient. A development may remain
internally consistent while drifting away from the original meaning of the
project. This document therefore separates:

- the intended mathematical claim;
- the current proved claim;
- the hypotheses currently required;
- the main refinements introduced during refactoring;
- the remaining alignment risks.

The note is meant to be reviewed manually and amended until the statement below
matches the intended Kairos story.

## 1. Intended Claim

The intended claim of the project is:

- given a reactive program `P`,
- an assumption automaton `A` describing admissible input streams,
- a guarantee automaton `G` describing acceptable input/output traces,
- structural well-formedness conditions ensuring that the program belongs to
  the total fragment modeled here,
- and a proof-producing toolchain that extracts local clauses from `P`, `A`,
  and `G`, and builds relational Hoare triples from them,

if the generated Hoare triples are valid,
then every admissible input stream should produce an execution whose trace
satisfies the guarantee.

Equivalently:

- if the generated Hoare triples built from the generated clauses are valid,
- if the program satisfies the accepted structural well-formedness conditions,
- if the chosen initial memory belongs to the admissible initial-memory
  predicate of the model,
- if the input stream avoids `bad_A`,
- then the program execution on this input avoids `bad_G`.

At this level, the claim is about:

- input streams;
- program executions;
- the trace generated by the program;
- safety acceptance of `A` and `G`.
- the generated local clauses extracted from the product;
- the relational Hoare triples built from them;
- and the validity of the corresponding proof objects.

What is still intentionally abstract here is the concrete implementation of
external validation:

- exact encoding of the generated Hoare triples/bundles;
- Why3 encoding;
- proof search;
- backend instrumentation details.

These latter notions should refine the proof architecture, without changing the
intended top-level semantic claim.

## 2. Current Mathematical Statement

The current end theorem is still the expected one.

At the final integration layer, we prove a conditional theorem of the form:

- `AvoidA u -> AvoidG (run_trace u)`
  (more explicitly: for every admissible initial memory `m0`,
  `AvoidA u -> AvoidG (run_trace_from m0 u)`)

under the following global proof assumptions:

- the program satisfies the named well-formedness predicate of the model;
- the chosen initial memory is admissible for the model;
- every generated triple is semantically valid;

See:

- [rocq/integration/AutomataFinalCorrectness.v](/Users/fredericdabrowski/Repos/kairos/rocq/integration/AutomataFinalCorrectness.v)
- [rocq/KairosOracle.v](/Users/fredericdabrowski/Repos/kairos/rocq/KairosOracle.v)

This is a good sign: the semantic conclusion has not changed into something
strictly weaker such as:

- correctness only for already-proved coherence facts;
- correctness only for a subset of traces;
- correctness only for the product IR itself.

The theorem still speaks about the program trace under admissible inputs, but
it is explicitly conditional on both structural well-formedness of the modeled
program, admissibility of the initial memory, and validity of the generated
relational Hoare triples.

An intermediate proof step then derives, by triple coverage, that the
generated clauses carried by these valid triples hold on the corresponding
concrete ticks. This derived fact is part of the proof architecture, not an
extra theorem hypothesis.

## 3. Current Proof Shape

The current proof architecture is now:

1. semantic product `program × A × G`;
2. extraction of generated semantic clauses from product steps;
3. construction of relational Hoare triples from these clauses;
4. validity of those triples;
5. recovery of semantic validity of the generated clauses;
6. reduction from a global violation of `G` to a locally violated generated
   clause.

Any later grouping of triples into transition-level Hoare-style bundles belongs
to the backend architecture, not to the intended mathematical core under audit.

This is a proof-method refinement, not a theorem change, provided each step is
conservative.

## 4. Conservative Refinements Already Introduced

The following refactorings appear conservative with respect to the intended
claim.

### 4.1 Explicit product state

We now reason explicitly over `program × A × G`.

This should be understood as:

- a semantic witness structure used to localize violations,
- not a change in the external meaning of program correctness.

Verdict:

- aligned with the intended claim.

### 4.2 `Clause` instead of `Obligation`

We renamed semantic local predicates from `Obligation` to `Clause`.

This is a clarification:

- a generated local predicate is not yet the external proof obligation;
- the external proof obligation is a Hoare-style bundle attached to a program
  transition.

Verdict:

- definitely aligned;
- reduces the risk of mistaking proof artifacts for semantic statements.

### 4.3 Hoare bundles per program transition

We now distinguish:

- generated semantic clauses;
- external transition-level bundles.

This is a change in the proof architecture, but not in the theorem being
proved, provided the bridge is sound:

- valid triples/bundles must imply validity of the covered clauses.

Verdict:

- aligned in principle;
- requires scrutiny of the bridge assumptions.

### 4.4 Coherence split

We now organize local reasoning into:

- `Safety / NoBad`;
- `Coherence / InitGoal`;
- `Coherence / Propagation`.

And within the coherence side:

- user invariants;
- automaton-support facts;

are treated together inside the same proof bundles.

This changes the proof organization, not the intended claim.

Verdict:

- aligned, and more realistic than the earlier flat four-way view.

### 4.5 Node invariants no longer postulated globally

Earlier, the Rocq core assumed node invariants globally true along the run.
They are now derived from:

- initial coherence clauses;
- propagation coherence clauses;
- validity of the generated coherence triples.

This makes the proof stronger, not weaker.

Verdict:

- aligned and strictly better.

## 5. Places Where Drift Is Still Possible

The main risks are no longer about the theorem statement itself, but about the
assumptions under which it is proved.

### 5.1 Totality of program ticks

The current model assumes that each tick produces:

- a next control state;
- a next memory;
- a current output.

This is stronger than arbitrary concrete Kairos programs, which may:

- have no active transition;
- be blocked by invalid operations;
- fail to emit on a signal.

So the current theorem is really a theorem about the fragment of Kairos
programs satisfying these totality assumptions.

Verdict:

- potentially stronger than the naive intended claim;
- acceptable only if we explicitly accept that the formalized object is the
  total, well-formed fragment.

The right presentation in the kernel is therefore not a collection of
scattered hypotheses, but a named predicate such as
`WellFormedProgramModel`.

### 5.1bis Initial memory admissibility

The current kernel no longer fixes a single known initial memory. Instead, the
theorem quantifies over an arbitrary initial memory `m0` satisfying the
predicate `init_mem_ok`.

This matches the intended Kairos reading much better:

- the control state is fixed at initialization;
- some ghost or monitor components may also be fixed;
- the rest of the memory may remain unknown.

Verdict:

- aligned with the intended claim;
- and strictly better than the earlier fixed-initial-memory model.

### 5.2 Deterministic transition selection

The formal core uses a deterministic selector `prog_select`.

If the concrete language admits several simultaneously active transitions, then
we are not proving correctness of the raw language semantics, but of:

- a determinized or priority-based interpretation of it.

Verdict:

- alignment depends on the intended execution model of Kairos;
- must be settled explicitly.

### 5.3 History interpretation

The intended claim is about specifications that may refer to the past.

The current mathematical story interprets such predicates over trace-derived
contexts, which is correct in spirit. But the implementation currently supports
only a specific fragment of history operators, compiled through auxiliary
variables.

Verdict:

- top-level claim still aligned;
- expressivity alignment between abstract model and implementation remains a
  real audit item.

### 5.4 External checker assumptions

The current kernel theorem does not model an external checker.

Its only proof-side assumption is:

- validity of the generated relational Hoare triples.

Any future modeling of Why3, solver answers, or validator soundness belongs to
an optional later refinement layer, not to the current audited theorem.

Verdict:

- aligned with the currently agreed intention.

## 6. What We Are *Not* Proving

To avoid accidental overclaiming, the current development does **not** yet
prove:

- completeness or soundness of Why3 itself;
- that every concrete Kairos program belongs to the total fragment;
- that every syntactically accepted history formula is compiled correctly in
  full generality;
- that every emitted Why3 verification condition is individually traced back in
  Rocq to a specific clause witness with complete implementation-level
  metadata.

So the current theorem should not be sold as:

- “fully verified end-to-end correctness of the whole Kairos toolchain”.

It is closer to:

- “a mechanized conditional reduction from global automaton correctness to
  validity of generated relational Hoare triples over a well-formed program
  model”.

## 7. Current Alignment Verdict

At this point, my best technical assessment is:

- the current theorem is still the intended theorem at the semantic level;
- the major refactorings mostly clarified the proof method and removed earlier
  unsound shortcuts;
- the main remaining risks are concentrated in the model boundary:
  - totality/determinism of program ticks;
  - expressivity of history;
  - exact scope of the accepted well-formedness conditions.

So I would not say “we are proving something different”.

I would say:

- we are still proving the right shape of theorem,
- but for a more explicitly delimited semantic model than the project may have
  originally stated informally.

## 8. Questions to Settle Explicitly

To decide whether the current development matches the intended project, we
should answer the following questions explicitly.

1. Is the intended object of study:
   - all Kairos programs,
   - or only total, well-formed ones?

2. Is the intended program semantics deterministic?
   - If yes, by what rule?
   - If not, the current selector-based model is too narrow.

3. For the current kernel, should the theorem mention an external proof
   backend?
   - Current answer: no.
   - The kernel should speak only about validity of generated relational
     Hoare triples.
   - External validation belongs to a later refinement layer.

4. Is the intended specification language abstractly history-based, with
   `prev` only as one instance?
   - If yes, the paper/model direction is correct.
   - If not, we should narrow the statement.

5. Is the intended tool claim:
   - semantic reduction to valid generated triples over a well-formed program
     model,
   - or a completely formalized implementation proof?

These answers should drive the next refactoring decisions.

## 9. Recommended Next Step

The useful next step is not code.

It is to review this note and mark each section as one of:

- `agreed`;
- `too strong`;
- `too weak`;
- `wrong model`.

Once those points are settled, we can decide whether the next effort should
target:

- the model boundary (totality, determinism, history fragment);
- a later external-validation bridge;
- or the project claim itself in the paper.
