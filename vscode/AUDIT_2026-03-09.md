## Detail par zone

### Extension VS Code
- Architecture actuelle:
  - un seul fichier `src/extension.ts`;
  - etat global minimal;
  - `any` utilise sur les structures de protocole et d'UI.
- Fonctionnalites presentes:
  - demarrage du client LSP;
  - trois vues arbre: Outline, Goals, Artifacts;
  - ouverture de documents virtuels pour:
    - `OBC+`,
    - `Why`,
    - `VC`,
    - `SMT`,
    - `DOT`,
    - `Labels`,
    - automates textuels,
    - `Obligations map`,
    - `Prune reasons`;
  - calcul local de fallback pour l'arbre des goals;
  - rendu PNG brut en webview.
- Lacunes structurelles:
  - absence de modules dedies;
  - aucune persistence de session;
  - aucun controle robuste des runs concurrents;
  - pas de barre de statut ni de modelisation forte des etats.

### Serveur LSP
- Capacites deja disponibles et utiles:
  - standard LSP:
    - diagnostics,
    - hover,
    - definition,
    - references,
    - completion,
    - document symbols,
    - formatting;
  - extensions Kairos:
    - outline,
    - goals tree final/pending,
    - instrumentation,
    - OBC,
    - Why,
    - obligations,
    - eval,
    - conversion DOT -> PNG,
    - run complet avec notifications `outputsReady`, `goalsReady`, `goalDone`;
  - support de `$/cancelRequest`.
- Limites actuelles cote produit:
  - aucun RPC de reset/session recent;
  - pas d'export structurel dedie;
  - pas de format SVG pour les graphes;
  - pas de notification riche de changement d'etape de pipeline autre que
    `$/progress` + notifications goals.

### Protocole
- Points positifs:
  - structures OCaml deja centralisees pour `outputs`, `automata_outputs`,
    goals, outline, config;
  - toutes les sorties necessaires existent deja pour une grosse partie de la
    refonte.
- Limites:
  - pas de types explicites pour un rendu de graphe interactif plus riche;
  - pas de type pour historique de run, export, ou resume de pipeline.

### UI native
- Points de reference a atteindre:
  - fenetre Eval avec edition, open/save et resultat lisible;
  - fenetre Automates avec quatre graphes;
  - barre de statut detaillee et journal local;
  - annulation, reset, cache, restauration partielle;
  - progression des goals et regroupement par noeud/transition;
  - diff d'Abstract Program.
- Limites de l'UI native a depasser dans VS Code:
  - rendu graphes encore fonde sur image raster;
  - pas de pipeline de rendu interactif DOM;
  - pas d'integration native VS Code possible evidemment.

## Ecarts UX/UI prioritaires

1. Automata: l'extension n'a pas de veritable studio d'exploration.
2. Eval: l'extension ne permet pas un travail iteratif serieux sur les traces.
3. Observabilite: absence de barre de statut, historique, chronometrie et
   signaletique de run.
4. Artifacts: la vue actuelle n'est pas une vue de travail synchronisee.
5. Preferences: l'extension n'expose presque rien de ce que le backend sait
   deja consommer.
6. Typage: l'usage de `any` empache la fiabilite du client.
7. Integration editeur: pas de code lenses, pas de menus contextuels, pas de
   restauration de session.

## Priorisation implementation

1. Refonte de l'architecture TypeScript et typage de bout en bout.
2. Etat applicatif unifie + status bar + historique local + annulation/reset.
3. Panels webview:
   - Automata,
   - Proof dashboard,
   - Artifacts workspace,
   - Eval playground.
4. Commandes, code lenses, menus, raccourcis et preferences VS Code.
5. Extensions LSP/protocole strictement necessaires au rendu/export final.
